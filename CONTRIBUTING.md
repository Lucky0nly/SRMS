# Contributing to SRMS\n\nThank you for your interest in contributing to the Student Record Management System (SRMS)! This document provides guidelines and instructions for contributing to this project.\n\n## Table of Contents\n\n- [Code of Conduct](#code-of-conduct)\n- [How to Contribute](#how-to-contribute)\n- [Development Setup](#development-setup)\n- [Coding Standards](#coding-standards)\n- [Testing](#testing)\n- [Commit Messages](#commit-messages)\n- [Pull Requests](#pull-requests)\n- [Reporting Bugs](#reporting-bugs)\n- [Suggesting Enhancements](#suggesting-enhancements)\n\n---\n\n## Code of Conduct\n\nThis project adheres to a Code of Conduct. By participating, you are expected to:\n\n- Use welcoming and inclusive language\n- Be respectful of differing opinions and experiences\n- Accept constructive criticism gracefully\n- Focus on what is best for the community\n- Show empathy toward other community members\n\n---\n\n## How to Contribute\n\n### Ways to Contribute\n\n1. **Report Bugs**: Help identify issues and edge cases\n2. **Suggest Features**: Propose improvements and new functionality\n3. **Write Documentation**: Improve or create documentation\n4. **Submit Code**: Fix bugs or implement new features\n5. **Code Review**: Review pull requests from other contributors\n\n---\n\n## Development Setup\n\n### Prerequisites\n\n- GCC or Clang compiler\n- Git\n- POSIX-compliant system (Linux, macOS, WSL)\n\n### Steps\n\n1. **Fork the repository**\n   ```bash\n   # Visit https://github.com/aarushdubey/SRMS and click \"Fork\"\n   ```\n\n2. **Clone your fork**\n   ```bash\n   git clone https://github.com/YOUR_USERNAME/SRMS.git\n   cd SRMS\n   ```\n\n3. **Add upstream remote**\n   ```bash\n   git remote add upstream https://github.com/aarushdubey/SRMS.git\n   ```\n\n4. **Create a feature branch**\n   ```bash\n   git checkout -b feature/your-feature-name\n   ```\n\n5. **Build the project**\n   ```bash\n   gcc -Wall -Wextra -O2 -o srms main.c\n   ```\n\n6. **Test your changes**\n   ```bash\n   ./srms\n   ```\n\n---\n\n## Coding Standards\n\n### C Programming Guidelines\n\n1. **Follow ANSI C Standards**\n   - Use standard C89/C99 features\n   - Avoid compiler-specific extensions\n\n2. **Naming Conventions**\n   - Use `snake_case` for variables and functions\n   - Use `UPPERCASE` for constants and macros\n   - Use descriptive names (e.g., `validate_input` not `val_inp`)\n\n3. **Code Formatting**\n   - Use 4 spaces for indentation (not tabs)\n   - Keep lines under 100 characters\n   - Use braces for all control structures\n   ```c\n   // Good\n   if (condition) {\n       function_call();\n   }\n   \n   // Bad\n   if (condition)\n       function_call();\n   ```\n\n4. **Comments and Documentation**\n   - Add comments for complex logic\n   - Use clear, concise language\n   - Document function purpose and parameters\n   ```c\n   // Hash password with salt using SHA-256\n   // Parameters:\n   //   salt - 16-character salt string\n   //   password - user password\n   //   output - 64-byte buffer for hash\n   void hash_password(const char *salt, const char *password, char *output) {\n       // Implementation\n   }\n   ```\n\n5. **Error Handling**\n   - Always check return values\n   - Validate input parameters\n   - Provide meaningful error messages\n\n6. **Security Best Practices**\n   - Use `fgets` instead of `gets` or `scanf`\n   - Validate and sanitize all input\n   - Avoid buffer overflows\n   - Use secure string handling functions\n\n---\n\n## Testing\n\n### Manual Testing\n\n1. **Test All Features**\n   - Test CRUD operations with valid data\n   - Test with invalid/boundary data\n   - Test authentication with wrong credentials\n   - Test role-based access control\n\n2. **Test Edge Cases**\n   - Empty input\n   - Very long input strings\n   - Special characters\n   - File corruption scenarios\n\n3. **Test Security**\n   - Verify password hashing\n   - Test atomic file operations\n   - Validate input sanitization\n\n### Automated Testing\n\nWhen implementing features, consider adding test cases:\n\n```bash\n# Create a test input file\necho -e \"2\\n101\\nJohn Doe\\n85.5\\n5\" | ./srms\n```\n\n---\n\n## Commit Messages\n\n### Format\n\nUse clear, descriptive commit messages:\n\n```\n<type>: <subject>\n\n<body>\n\n<footer>\n```\n\n### Types\n\n- **feat**: A new feature\n- **fix**: A bug fix\n- **docs**: Documentation changes\n- **refactor**: Code refactoring\n- **test**: Adding tests\n- **chore**: Maintenance tasks\n\n### Examples\n\n```\nfeat: Add user profile management\n\nImplement ability for users to update their profile information\nincluding name, email, and preferences.\n\nCloses #42\n```\n\n```\nfix: Fix buffer overflow in student name input\n\nUse fgets instead of scanf for safer input handling.\nValidate input length before processing.\n```\n\n---\n\n## Pull Requests\n\n### Before Submitting\n\n1. **Update your branch**\n   ```bash\n   git fetch upstream\n   git rebase upstream/main\n   ```\n\n2. **Test thoroughly**\n   - Compile without warnings\n   - Test all code paths\n   - Verify no regressions\n\n3. **Clean up commits**\n   - Squash related commits\n   - Write clear commit messages\n\n### Submitting a PR\n\n1. **Push to your fork**\n   ```bash\n   git push origin feature/your-feature-name\n   ```\n\n2. **Create pull request on GitHub**\n   - Use a clear, descriptive title\n   - Reference related issues\n   - Describe changes clearly\n   - Include test results\n\n### PR Description Template\n\n```markdown\n## Description\nBrief description of changes\n\n## Type of Change\n- [ ] Bug fix\n- [ ] New feature\n- [ ] Documentation\n- [ ] Refactoring\n\n## Testing\nDescribe testing performed\n\n## Checklist\n- [ ] Code compiles without warnings\n- [ ] Code follows style guidelines\n- [ ] New features documented\n- [ ] No breaking changes\n- [ ] Tested with various inputs\n```\n\n### Review Process\n\n1. Maintainers will review your PR\n2. Address feedback and make requested changes\n3. Keep discussions professional and constructive\n4. Once approved, your PR will be merged\n\n---\n\n## Reporting Bugs\n\n### Bug Report Guidelines\n\n1. **Check existing issues** to avoid duplicates\n2. **Use a clear title** that describes the issue\n3. **Provide detailed description**\n4. **Include reproduction steps**\n5. **Share expected vs. actual behavior**\n6. **Provide system information**\n\n### Bug Report Template\n\n```markdown\n## Description\nClear description of the bug\n\n## Steps to Reproduce\n1. Step 1\n2. Step 2\n3. ...\n\n## Expected Behavior\nWhat should happen\n\n## Actual Behavior\nWhat actually happens\n\n## System Information\n- OS: [e.g., Ubuntu 20.04]\n- Compiler: [e.g., GCC 9.3.0]\n- SRMS Version: [e.g., v1.0.0]\n\n## Additional Context\nAny other relevant information\n```\n\n---\n\n## Suggesting Enhancements\n\n### Enhancement Suggestions\n\n1. **Use a clear title**\n2. **Describe the problem** you're trying to solve\n3. **Suggest a solution** (if you have one)\n4. **Include examples** or use cases\n5. **Consider alternatives**\n\n### Enhancement Template\n\n```markdown\n## Problem\nDescription of the current limitation\n\n## Suggested Solution\nHow you would like to see it improved\n\n## Alternative Solutions\nOther approaches considered\n\n## Use Cases\nScenarios where this would be helpful\n```\n\n---\n\n## Questions?\n\n- Check the README.md for project documentation\n- Review existing issues and discussions\n- Open a new discussion for questions\n\n---\n\n## Recognition\n\nContributors will be recognized in the project's README and release notes. Thank you for helping make SRMS better!\n\n---\n\n**Last Updated**: December 2025
